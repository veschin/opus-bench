# opus-bench

Daily canary monitoring for Anthropic's Claude Opus models, run through the
`claude` CLI to mirror what the user actually invokes (system prompt + plugins
+ MCP servers included in every call).

## What it measures

Compares Claude Opus 4.6 and 4.7 across six tasks (`reason`, `trace`, `code`,
`bugs`, `behave`, `refusal`) with `K=3` repeats per (task × model). Three
scoring layers per response:

1. **Programmatic correctness** - exact match for math/trace, executed unit
   tests for code.
2. **Sonnet polyrubric** - a single Sonnet-4.6 call evaluates ten axes per
   response:
   - `correctness`, `depth` - higher is better (0..10)
   - eight defect axes - `servile`, `overconfident`, `lazy`, `noise`,
     `hedging`, `drift`, `shallow`, `hallucination` - higher is worse
   - evidence quote required when defect score >= 4
3. **Thinking effort proxies** - derived from the raw API response:
   - `thinking_tokens_proxy = max(0, output_tokens - len(visible_text)/3.5)`
   - `duration_api_ms`
   - `thinking_blocks` (count of thinking content blocks, including the
     encrypted ones returned by Opus 4.7)

A final Sonnet call reads the aggregates and emits a single
`USE: <model>` verdict with two or three bulleted reasons citing concrete
metrics.

## Why this design

Earlier versions repeated three failure modes:

- v1 measured `len(block.thinking)` directly. Opus 4.7 returns thinking
  encrypted (populated `signature`, empty `thinking` text), so the bench
  read 0 and pinned 4.7 at 40% DEGRADED while the actual answers were
  correct on every task.
- v1 scored bug-finding and diagnostic tasks via English-only regex.
  Russian-language responses missed all matches and lost points purely
  for language choice.
- v2 dropped the thinking metric and scored each task pass/fail. Both
  models hit 6/6 on a six-task suite, so a known degradation between
  them was invisible.

v3 fixes these by:

- using a multi-axis Sonnet judge that processes both English and Russian
  responses by content rather than regex,
- treating `output_tokens − visible_text` as the thinking-effort proxy, which
  captures encrypted thinking too,
- producing continuous metrics rather than thresholds, with the final
  verdict generated by a Sonnet analyst from the aggregated JSON.

## Requirements

Three binaries on PATH:

- `claude` - Anthropic's Claude Code CLI (used for both target Opus calls
  and the Sonnet judge). Install: <https://docs.claude.com/en/docs/claude-code>.
- `gum` - Charmbracelet's TUI helper (used for the spinner and styled
  panels). Install: `brew install charmbracelet/tap/gum`, or see
  <https://github.com/charmbracelet/gum>.
- `python3` - for the inline scoring/aggregation block (no third-party
  packages needed; standard library only).

Quick check that everything is wired up:

```sh
for c in claude gum python3; do command -v "$c" >/dev/null && echo "ok  $c -> $(command -v $c)" || echo "MISSING  $c"; done
```

You also need an Anthropic account with sufficient quota for both Opus
and Sonnet - the bench makes about 73 API calls per full run.

## Usage

```sh
./opus-bench               # full run, short verdict only
./opus-bench --verbose     # full run, all tables + verdict
./opus-bench --fast        # K=1 smoke
./opus-bench --fast -v     # smoke + tables
./opus-bench --help
```

Cost per full run: roughly $10-14 (36 Opus + 36 Sonnet polyrubric +
1 analyst).
Wall time: 7-10 minutes with parallelism.

## Output

- `results/bench_<DATE>.json` - aggregated metrics per model plus the
  analyst verdict text.
- `results/<task>_<model>_<DATE>_r<N>.json` - raw API response for each
  (task, model, run).

A trend across recent v3 runs is shown in `--verbose` mode.

## Limitations

- Measures CLI inference, not the raw model. System prompt, plugins, and
  MCP servers are part of every call. A regression in those layers will
  be attributed to the model.
- The same Sonnet judge evaluates both target models, so judge bias is
  uniform between them. Cross-model comparisons are valid; absolute
  defect levels are not.
- `K=3` smooths variance but does not provide statistical significance
  for differences below roughly 10 percentage points. For drift detection
  across days, treat the JSON history as a control chart, not a
  single-shot verdict.
- `claude -p --tools ""` does not actually disable MCP servers - they
  remain in the system prompt (about 16K tokens of `cache_creation` per
  call). This inflates per-call cost but does not bias model comparison.

## Files

- `opus-bench` - main script (bash with inline Python).
- `results/` - historical run artifacts (raw JSON per call plus
  per-run aggregate).

## Why not Inspect AI / promptfoo / Aider polyglot

Considered and rejected as overkill for a daily one-host canary:

- Inspect AI and promptfoo target reusable evaluation infrastructure;
  this bench is a single-purpose script that lives in one shell file.
- SWE-Bench Verified is too expensive (50+ USD) and slow (hours) for
  daily cadence.
- Aider polyglot benchmark needs Docker spin-up and exceeds the
  10-minute window.

If the bench grows beyond ad-hoc daily monitoring, Inspect AI is the
natural migration path.
